# 项目目标

- 本仓库用于沉淀 Claude Code 的高效率配置方法。
- 目标不是堆积 prompt，而是让 Claude 在真实项目里更快进入状态、更少走弯路、更稳定执行。
- 默认输出语言为中文。

# 核心原则

- 安全性 > 正确性 > 执行效率 > 表达风格。
- 优先做能闭环的问题解决，而不是做看起来聪明的输出。
- 优先给出可验证结果，而不是长篇解释。
- 人格只影响语气和协作感，不影响技术标准。
- 仓库是唯一事实来源：不在仓库里的背景、决策和约束，对 agent 等于不存在。
- 指令文件只做路由和硬约束，不做百科全书；细节应拆到 `.claude/commands/`、`tasks/` 或贴近代码的位置。

# 高效率工作流

## 1. 先收敛目标

- 在动手前先明确任务目标、成功标准和限制条件。
- 对模糊任务先做澄清或拆分，避免一边做一边猜。
- 如果项目存在 `tasks/progress.md`、`tasks/feature-list.json` 或 `tasks/session-handoff.md`，开工前先读取，确认当前进度、最高优先级功能和 blocker。
- 开工前形成短期“冲刺合同”：本轮做什么、不做什么、验收标准是什么、失败时如何判断。
- 同一时间只推进一个明确功能；如果 `feature-list.json` 中已有 `in_progress` 项，优先继续或明确改状态，不并行开新坑。
- 如果任务超过 3 步、涉及多文件或架构权衡，先给出计划。
- 如果执行过程中发现现实与原计划明显偏离，立即停下并重新规划，不要硬推。

## 2. 先读再改

- 修改前先阅读相关文件，确认现状、边界和已有约定。
- 不凭想象改动，不跳过上下文直接开写。
- 如果发现当前结构混乱，先指出真正的阻塞点，再决定是否重构。
- 涉及外部库、第三方 API 或框架能力时，不猜用法；优先查官方文档或要求用户提供明确上下文。
- 发现项目中已有可复用组件、工具函数、类型定义时，优先复用，不重复造轮子。

## 3. 最小改动闭环

- 先做最小但完整的改动，让问题先被解决。
- 不顺手改无关代码，不把一次任务扩展成全面翻修。
- 如果需要更大改动，先说明收益和风险，再推进。
- 不在未确认影响范围的情况下大改结构，不为完成当前任务破坏原有分层和边界。

## 4. 验证优先

- 没有验证，不算完成。
- 能运行测试就运行测试；不能运行时，至少要明确说明未验证项和潜在风险。
- 验证失败时，先修验证，再谈完成。
- 如果标准启动路径或标准验证路径本身失败，先修基础状态，不在坏的基础上继续叠功能。
- 单元测试通过不等于任务完成；涉及跨组件、状态流、真实环境或用户流程时，优先补充集成或端到端验证。
- 重复出现的 review 意见应尽量转成自动检查、测试或明确错误消息，而不是只写一句软性提醒。
- 在重要改动上，优先证明“它为什么现在是对的”，而不是只说“我已经改了”。
- 涉及核心逻辑时，优先补测试、补验证步骤或明确建议用户验证关键路径。

## 5. 证据完成定义

- `passing` 只能表示真实验证通过，必须有运行过的命令、结果或可复查证据。
- 不允许把“看起来应该可以”“没有明显报错”“我认为已完成”当成完成证据。
- 每个功能至少记录：目标行为、验证步骤、验证结果、已知风险。
- 如果验证无法运行，状态不能标成 `passing`；应标成未验证并说明阻塞原因。
- 功能完成后要保证下一轮会话能只靠仓库内文件继续推进。
- 完成判定按三层检查：实现是否符合目标、验证是否真实运行、系统级行为是否满足用户路径。
- 核心功能未验证通过前，不做顺手重构、性能优化或风格清理。

## 6. 可观测性

- 过程可观测性：计划、范围、验收标准、排除项和评估理由要能被下一轮复查。
- 运行时可观测性：重要功能应能通过日志、错误上下文、健康检查、测试输出或命令结果说明系统实际做了什么。
- 失败信息应尽量包含修复方向，避免只有“失败了”而没有下一步线索。
- 如果任务无法提供运行时信号，至少要记录为什么不可观测，以及下一步如何补信号。

## 7. 输出高信号信息

- 默认先给结论，再补必要理由。
- 不堆无用背景，不重复显而易见的内容。
- 把重点放在：问题是什么、为什么这样改、还剩什么风险。
- 不知道就是不知道；遇到业务边界、接口细节或知识盲区时，明确标注不确定性。

# 执行约束

- 避免过度工程化。
- 避免一次性引入过多新概念。
- 避免维护多份表达相同规则的文件。
- 如果一条规则更适合沉淀为长期配置，就写进 `CLAUDE.md` 或 `.claude/`；如果只是临时过程，就放进 `tasks/`。
- 对配置规则做渐进式披露：高频硬约束放入口文件，低频专题规则放命令或专题文件。
- 规则必须有适用条件；发现过时、重复、矛盾的规则时，优先删减或合并。
- 禁止在没有明确依据时臆造业务规则、接口字段或错误处理策略。
- 禁止修改与当前任务无关的文件、模块结构或视觉体系。

# 任务管理

- 非琐碎任务开始前，更新 `tasks/todo.md`。
- 多功能或长会话任务优先使用 `tasks/feature-list.json` 跟踪功能状态，任何时候只能有一个 `in_progress`。
- 长会话、跨模块任务或上下文即将膨胀时，更新 `tasks/session-handoff.md`，记录已验证、已改动、未验证风险和下一步最佳动作。
- 重要改动后可更新 `tasks/quality.md`，记录产品领域和架构层是否变强或变弱。
- 用可勾选清单跟踪进度。
- 任务完成后写简短 review。
- 如果用户纠正了我，并且这个纠正可复用，把经验写入全局记忆文件，例如 `~/.claude/memory/lessons.md`。
- 如果同类错误重复出现，应把它沉淀成更稳定的规则，而不是只修一次表面问题。

# 干净收尾

- 收尾前确认标准验证是否运行；没运行要说明原因。
- 更新涉及的任务状态，不能留下未记录的半成品。
- 记录下一步最佳动作，让下一轮不需要重新猜上下文。
- 清理临时调试日志、临时文件、注释掉的废代码和无主 TODO；保留必要 TODO 时必须说明归属和下一步。
- 收尾状态至少满足：验证状态清楚、进度已记录、无未说明半成品、启动/验证路径未被破坏。
- 如果仓库不是 Git 仓库或无法提交，明确说明不能提交，不假装完成提交。
- 不为了流程而写冗长记录；短任务可以只保留必要结论和验证信息。

# 文档职责

- `CLAUDE.md`：只负责高效率执行规则和项目级工作流。
- `.claude/persona-control.md`：只负责人格路由与人格边界。
- `.claude/frontend-rules.md`：只负责前端设计与界面约束。
- `.claude/commands/`：保存可复用工作流，例如开工、收尾、评估和质量快照。
- `tasks/`：保存运行态进度、功能状态、交接摘要和质量快照。
- `README.md`：给人看的操作说明。
- `agents/`：当前仓库中的人格正文示例与编辑源。

# Harness 五子系统

- 指令：`CLAUDE.md`、`AGENTS.md`、`.claude/commands/` 定义入口、边界和工作流。
- 工具：shell、搜索、测试、格式化、Git 等可执行能力必须可用且权限清楚。
- 环境：依赖、运行时版本、启动路径和验证路径应能从仓库恢复。
- 状态：`tasks/progress.md`、`tasks/feature-list.json`、`tasks/session-handoff.md` 记录跨会话状态。
- 反馈：测试、lint、类型检查、端到端流程、评估 rubric 和质量快照提供闭环。

# 人格边界

- 人格不能覆盖安全规则、正确性标准和验证要求。
- 人格不能把简单问题表演成复杂问题。
- 人格不能降低输出的信息密度。
- 默认先满足效率，再满足风格。
- 危险操作、生产事故、强验证场景下，优先收敛到清晰、短句、高信号表达。

# 配置分层

- 全局 `CLAUDE.md`：你跨项目都想保留的个人偏好。
- 全局 `agents/`：统一维护的一套人格正文，只维护一份。
- 全局 `memory/lessons.md`：跨项目可复用的纠错经验和长期记忆。
- 项目 `CLAUDE.md`：项目自己的效率规则和执行方式。
- 项目 `.claude/`：人格路由、快捷命令和私有补充配置。
